# Repository Guidelines

## プロジェクト構成とモジュール整理
`lib/` に Flutter アプリ本体のコードがあります。現在のエントリーポイントは `lib/main.dart` です。画面、状態管理、ドメインロジックが増えてきたら、`main.dart` に集約せず、機能単位のフォルダへ分割してください。

`test/` には自動テストを置きます。新しいテストは `lib/` の構成に合わせて配置してください。プラットフォーム固有の設定とランナーは `android/` と `ios/` にあります。`document/` には事業計画、Firestore 設計、機能設計、画面モックなどの設計資料を保存しています。特に `document/02_FDD/` は機能設計、`document/03_designs/` はブラウザで確認できるデザインモックです。

## ビルド・テスト・開発コマンド
以下はリポジトリ直下で実行します。

- `flutter pub get`: Dart / Flutter の依存関係を取得します。
- `flutter run`: 接続中のシミュレータまたは実機でアプリを起動します。
- `flutter test`: `test/` 以下のユニットテストとウィジェットテストを実行します。
- `flutter analyze`: `flutter_lints` に基づく静的解析を実行します。
- `dart format lib test`: `lib/` と `test/` の Dart コードを整形します。

日常開発ではホットリロードを使って構いませんが、PR 作成前に `flutter analyze` と `flutter test` は必ず通してください。

## コーディング規約と命名
Dart の標準スタイルに従います。インデントは 2 スペース、整形しやすい箇所では末尾カンマを使い、1 つの公開クラスやウィジェットには 1 つの明確な責務を持たせてください。クラスとウィジェットは `UpperCamelCase`、メソッドや変数は `lowerCamelCase`、ファイル名は `snake_case.dart` を使用します。

ウィジェットは小さく分割し、再利用できる画面・モデル・ヘルパーは `lib/main.dart` から切り出してください。`analysis_options.yaml` の lint は基本的に守り、`ignore` を追加する場合は理由をその場に残してください。

## テスト方針
ウィジェットテストには `flutter_test` を使います。テストファイル名は `*_test.dart` とし、説明文は `testWidgets('opens book details', ...)` のように挙動ベースで書いてください。

ユーザーに見える変更には原則としてテストを追加または更新します。最低限、新しい画面遷移、ウィジェットの状態変化、修正した不具合の再発防止はカバーしてください。

## 設計資料の扱い
実装前後には `document/` の関連資料を確認してください。仕様変更や画面変更を伴う場合は、コードだけでなく設計書やモックも必要に応じて更新し、実装と資料の差分を残さないようにします。

## コミットとプルリクエスト
直近の履歴では `add flutter` や `Update README with design mock link` のような、短い命令形のコミットメッセージが使われています。件名は簡潔に、できれば 72 文字以内に収め、1 コミット 1 論理変更を基本にしてください。

PR には変更概要、関連 Issue またはタスク、実行したテスト結果、UI 変更時のスクリーンショットまたは録画を含めてください。`document/` を更新した場合は、その旨を明記してコードと仕様を一緒に確認できるようにしてください。
